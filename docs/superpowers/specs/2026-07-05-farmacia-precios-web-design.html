01

Contexto

El proyecto ya tiene un prototipo funcional (1_extraer_muestra.py a 4_generar_web.py) que toma una muestra de 150 productos, consulta precios en la API pública de Drogas La Rebaja, fusiona descripciones — reales o generadas por IA — y genera una página HTML estática autocontenida.

Este diseño define cómo escalar ese prototipo a producción real para las 3 droguerías del negocio.

02

Cada decisión de abajo salió de una conversación de calibración con el dueño del negocio, no de una suposición de diseño.

03

Arquitectura general

El cómputo pesado se mueve a la nube para no depender de que un computador de sede quede encendido de madrugada durante horas.

Computador local

1 sede · segundos, no horas

  1. Lee productos.xls
  2. Normaliza el catálogo
  3. Sube catalogo.json a git

GitHub Actions

Nube · cron diario fijo, ~3 AM

  1. Selecciona el lote del día
  2. Scraping vs. La Rebaja
  3. Fusiona descripciones
  4. Regenera datos.json + web
  5. Publica en GitHub Pages

3 puntos de venta

Misma URL fija

  1. Sede 1 — abre el link
  2. Sede 2 — abre el link
  3. Sede 3 — abre el link

El cron de la nube corre siempre a su hora, aunque el paso local no haya subido catálogo nuevo ese día.

Decisión clave

Todo el trabajo pesado corre en GitHub Actions, gratis e ilimitado por ser un repositorio público. El paso local dura segundos; nadie necesita dejar un computador encendido a las 3 AM para que el sistema funcione.

04

Cruce de datos de ventas

Para rankear qué productos son "más vendidos", se cruzó el catálogo real (16.810 productos) contra un histórico de ventas real (102.176 filas de facturación, ene–jun 2026, 3.432 productos únicos vendidos). Ningún campo de ID es compartido entre los dos archivos — el cruce se hizo por nombre normalizado.

19.2%
2.773 productos — coincidencia exacta por nombre 659 productos — sin coincidencia confiable

Por qué se descartó el fuzzy matching

Se probó coincidencia difusa para rescatar el 19.2% restante. Se descartó: generaba cruces incorrectos y potencialmente peligrosos en un contexto de droguería.

Decisión

Usar únicamente el cruce exacto por nombre normalizado. Lo que no cruza — vendido sin coincidencia, o nunca vendido en el periodo — se trata como "sin dato de ventas" y entra a la rotación pareja de la sección 5, sin forzar un match dudoso.

Métrica de ranking: suma de unidades vendidas (Cantidad Caja + Blister + Unidad) por producto en el periodo disponible.

05

Priorización y rotación de lotes

Scrapear los 16.810 productos completos tomaría ~2.8 horas (0.6s de pausa respetuosa por producto). Se dividen en dos niveles para bajar eso a minutos por día, sin dejar de cubrir el catálogo entero cada semana.

Tier · siempre ~1.500

Los más vendidos por unidades. Se actualizan todos los días, sin excepción.

Tier · rotación 15.310

El resto del catálogo, dividido en 7 franjas fijas y deterministas (ej. hash del EAN mod 7).

Presupuesto diario ~3.687

1.500 fijos + 1 franja de ~2.187. Dentro del objetivo de ~4.000/día.

Cada producto cae siempre en el mismo día de la semana — no hace falta guardar un cursor de estado entre ejecuciones. Si un día se salta el workflow, el ciclo se autocorrige solo la semana siguiente.

La web muestra la fecha del último precio de competencia consultado por producto, ya que no todos se refrescan el mismo día.

06

Manejo de errores y confiabilidad

  • Errores por producto: reintento con espera progresiva (2 intentos) ante fallos de red; si persiste, se conserva el último precio conocido — nunca se sobreescribe con null.
  • Escrituras atómicas: los archivos de datos se escriben a un temporal y se reemplazan al finalizar, para no dejar el catálogo corrupto ante un corte a mitad de proceso.
  • Resiliencia del pipeline: el workflow en la nube corre siempre a su hora fija, incluso si el paso local no subió catálogo nuevo ese día.
  • Protección contra publicaciones rotas: si la tasa de coincidencia con La Rebaja cae drásticamente frente al día anterior, se aborta esa publicación y se conserva la última versión buena conocida.
  • Notificaciones: aviso por correo nativo de GitHub Actions cuando un workflow programado falla — sin integraciones adicionales.
  • Registro: cada ejecución deja un resumen (actualizados, fallidos, sin match, duración) en el historial de Actions.
07

Diseño de la web

El HTML actual embebe todos los datos como JSON dentro del mismo archivo: 65KB para 150 productos. Con 16.810 productos serían ~7MB en un solo archivo, y renderizar todas las tarjetas de una vez sería lento en un computador de punto de venta.

  • Separar datos de la página: index.html liviano que descarga datos.json aparte, una sola vez.
  • Buscar en memoria, renderizar poco: el filtrado ocurre sobre los 16.810 productos en memoria, pero solo se muestran los primeros ~50 resultados coincidentes.
  • Sin listado inicial: se pide escribir un término de búsqueda antes de mostrar nada, en vez de listar el catálogo completo al cargar.
  • Sin caché vieja: datos.json lleva una marca de versión/fecha en la URL para que el navegador siempre traiga la publicación más reciente.
  • Se conserva el resaltado verde/rojo (precio propio menor/mayor que La Rebaja) y se añade la fecha del último precio consultado.
